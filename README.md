# n8n Infrastructure Automation on Microsoft Azure

This repository provides a production-grade Infrastructure as Code (IaC) implementation to deploy **[n8n](https://n8n.io/)** on Microsoft Azure. Architected entirely in **Terraform**, this solution emphasizes strict security boundaries, network isolation, and modular design principles suitable for enterprise environments.

---

## Architecture Overview

The infrastructure relies on a decoupled, modular architecture to ensure reusability and localized state management. The orchestration is defined in the root `main.tf`, which integrates the following components:

- **`network`:** Provisions the foundational Virtual Network (VNet) topology. Implements logical segmentation using targeted subnets (e.g., dedicated subnets for compute, database, and gateway components).
- **`compute`:** Deploys the Virtual Machine hosting the n8n workload. Configured with a System-Assigned Managed Identity to interact with Azure resources securely and without explicit credentials.
- **`database`:** Aprovision the backend relational database required for n8n's persistence layer, restricted to internal network access.
- **`security`:** Provisions an Azure Key Vault for centralized secrets management. Dynamically generates and securely stores sensitive data (e.g., database credentials).
- **`load_balancer`:** Configures inbound traffic distribution, ensuring the n8n application is exposed securely.
- **`dns`:** Automates the integration and resolution of domain name records.

---

## Security Posture

Security is integrated at the core of the deployment utilizing Azure's native IAM and Zero Trust models:

- **Identity-Based Access (RBAC):** Access to secrets is granted exclusively over Role-Based Access Control. The Compute node is granted `Key Vault Secrets User` strictly scoped to the provisioned Key Vault vault.
- **Zero-Touch Secrets:** Database credentials are autogenerated by Terraform, immediately persisted to Azure Key Vault, and never exposed in plaintext outputs or configuration files.
- **State File Integrity:** The deployment configuration strictly delegates the `.tfstate` files to the `.gitignore` policy, preventing accidental credential exposure to version control systems.

## Prerequisites

Ensure the deployment environment satisfies the following dependencies:

1. **[Terraform CLI](https://developer.hashicorp.com/terraform/downloads)** (v1.0 or later).
2. **[Azure CLI](https://docs.microsoft.com/en-us/cli/azure/install-azure-cli)** (Authenticated via `az login` with sufficient RBAC permissions, such as Contributor and User Access Administrator).
3. A designated public SSH key (e.g., `id_rsa_n8n.pub`) located in the root repository for authorized administrative access to the compute nodes.

## Deployment Operations

1. **Initialize the Working Directory**
   Prepare the backend and download the required AzureRM provider plugins:
   ```bash
   terraform init
   ```

2. **Define Environment Specifications**
   Override default parameters by creating a `terraform.tfvars` file or passing variables at runtime. Ensure critical access controls, such as `my_ip` in `main.tf`, are strictly scoped to trusted administrative networks.

3. **Generate Execution Plan**
   Validate the syntax and review the prospective infrastructure modifications:
   ```bash
   terraform plan
   ```

4. **Provision Infrastructure**
   Execute the deployment:
   ```bash
   terraform apply
   ```

---

## Maintenance and State Management

- **Remote Backend Recommendation:** While this repository defaults to a local state execution for rapid prototyping, it is **strongly advised** to migrate the Terraform state to an **Azure Storage Account** backend with Blob versioning and state locking configured prior to production workloads.
- **Credential Rotation:** Since secrets are managed by Azure Key Vault, they can be rotated seamlessly without requiring redeployment of compute assets, adhering to standard compliance regulations.
